# AIOps RAG 知识库维护指南

---

## 目录结构

所有知识文档存放在 `examples/aiops-docs/`（或你指定的自定义路径）：

```
aiops-docs/
├── services/                    # 服务文档
│   └── {service_id}-{name}/     # 目录名: ID+名称防重名
│       ├── README.md            # 服务简介
│       ├── tech-arch.md         # 技术架构
│       ├── sop-restart.md       # 重启 SOP
│       ├── sop-scale.md         # 扩容 SOP
│       └── sop-xxx.md           # 其他 SOP
├── hosts/                       # 主机文档
│   └── {host_id}.md
├── incidents/                   # 故障复盘
│   └── {YYYY-MM-DD}-{简述}.md
├── topology/
│   └── call-graph.yml          # 服务拓扑定义
└── shared/                      # 通用文档
```

## 一、添加新服务

### Step 1: 生成 service_id

每个服务分配一个**不可变唯一标识符**：`svc_xxxxx`

命名规则：
- 前缀 `svc_` + 服务简写（小写、英文、下划线分隔）
- 如：`svc_nginx`、`svc_order`、`svc_db_mysql`
- 一旦分配，永不修改（即使服务名变更）

### Step 2: 创建文档目录

```bash
mkdir -p aiops-docs/services/{service_id}-{service_name}/
```

例如：
```bash
mkdir -p aiops-docs/services/svc_api_gateway-api-gateway/
```

### Step 3: 创建 README.md

```markdown
---
title: {服务名} 简介
doc_type: tech
service_ids:
  - svc_api_gateway
service_name: api-gateway
tags:
  - 网关
  - 服务简介
related_services:
  - svc_order
  - svc_user
related_hosts:
  - host_proxy_01
author: 你的名字
updated_at: 2026-06-01
---

## {服务名}

### 功能概述

负责 API 路由和流量分发...

### 技术栈

- 语言: Go 1.22
- 框架: Gin
- 端口: 8080

### 负责人

- 开发: 张三
- 运维: 李四
```

### Step 4: 创建 SOP 文档

SOP 格式见 [第二节](#二sop-文档格式)。

### Step 5: 更新拓扑

在 `topology/call-graph.yml` 的 `services` 列表中追加新服务，写清 `deploys_on`、`calls`、`ports`。

格式见 [第四节](#四更新拓扑关系)。

### Step 6: 入库

```bash
docker exec rag-api python3 /app/scripts/index-docs.py /app/examples/aiops-docs/
docker exec rag-api python3 /app/scripts/load-topology.py /app/examples/aiops-docs/topology/call-graph.yml
```

## 二、SOP 文档格式

### 模板

## 标题

### 前置条件

- 需要哪些权限（如 root、sudo）
- 需要提前检查什么
- 影响范围（会影响哪些服务/用户）

### 操作步骤

#### 1. 步骤一

```bash
# 具体命令
systemctl status xxx
```

#### 2. 步骤二

...

### 验证

如何确认操作成功：
- 检查进程: `systemctl status xxx`
- 检查端口: `ss -tlnp | grep 8080`
- 检查日志: `tail -20 /var/log/xxx/app.log`

### 回滚

操作失败时如何恢复到操作前状态。

### 常见问题

| 现象 | 原因 | 处理 |
|------|------|------|
| 重启后端口未监听 | 配置错误 | 回滚配置后重启 |
```

## 三、Frontmatter 字段详解

每个 Markdown 文件头部的 YAML 区：

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `title` | string | ✅ | 文档标题（用于检索和展示） |
| `doc_type` | string | ✅ | 文档类型：`sop` / `tech` / `incident` |
| `service_ids` | list | ✅ | 关联的服务ID列表（至少1个，每个以 svc_ 开头） |
| `service_name` | string | ❌ | 关联的服务名（冗余，方便检索）|
| `tags` | list | ❌ | 标签，帮助搜索 |
| `related_services` | list | ❌ | 涉及的其他服务ID |
| `related_hosts` | list | ❌ | 涉及的主机ID |
| `author` | string | ❌ | 作者 |
| `updated_at` | string | ❌ | 最后更新日期 `YYYY-MM-DD` |

## 四、更新拓扑关系

### 文件位置

`aiops-docs/topology/call-graph.yml`

### 格式

```yaml
services:
  - id: svc_api_gateway          # 服务唯一ID
    name: api-gateway             # 服务名称
    deploys_on: host_proxy_01     # 部署在哪台主机（host id）
    calls:                        # 调用了哪些下游服务
      - target: svc_order
        protocol: http
        port: 8080
      - target: svc_user
        protocol: grpc
        port: 50051
    ports:                        # 服务监听的端口
      - 443
      - 80

hosts:
  - id: host_proxy_01            # 主机唯一ID
    name: prod-proxy-01           # 主机名/用途
    ip: 10.0.1.10                 # IP地址
    os: Ubuntu 22.04              # 操作系统
```

### 操作清单

#### 添加服务到拓扑

1. 在 `services:` 下追加一条新条目
2. 确保 `deploys_on` 的 host 已存在于 `hosts:` 列表中（不存在则先创建主机）
3. `calls` 中引用的 target 可以是已在 `services` 中的任何服务
4. 入库:
   ```bash
   docker exec rag-api python3 /app/scripts/load-topology.py /app/examples/aiops-docs/topology/call-graph.yml
   ```

#### 添加主机到拓扑

1. 在 `hosts:` 下追加新条目
2. 分配未使用的主机 ID（如 `host_db_02`）
3. 入库

#### 修改服务调用关系

1. 编辑对应服务的 `calls` 列表
2. 入库

#### 删除服务

1. 从 `services:` 列表移除
2. 移除所有引用该服务的 `calls` 中的 target
3. 删除对应文档目录
4. 入库

## 五、文档生命周期

### 添加文档

1. 在对应分类目录下创建 Markdown 文件
2. 按格式填写 frontmatter + body
3. 运行索引（增量，不删已有数据）：
   ```bash
   docker exec rag-api python3 /app/scripts/index-docs.py /app/examples/aiops-docs/
   ```

### 更新文档

1. 直接编辑 Markdown 文件（修改 body 或更新 `updated_at`）
2. 运行索引（同 `doc_id` 的文档会 upsert 覆盖，ID 由 `service_id` + `文件名` + `chunk_index` 决定）
3. 不需要删除旧版本

### 删除文档

1. 删除 Markdown 文件
2. 手动从 ES 删除对应 chunk：
   ```bash
   # 查找 doc 在 ES 中的 _id
   curl -s -X POST 'http://localhost:9200/knowledge_base/_search' \
     -H 'Content-Type: application/json' \
     -d '{"query": {"match": {"doc_id": "{service_id}_{filename}"}}}' | python3 -m json.tool

   # 批量删除
   curl -s -X POST 'http://localhost:9200/knowledge_base/_delete_by_query' \
     -H 'Content-Type: application/json' \
     -d '{"query": {"match": {"doc_id": "{service_id}_{filename}"}}}'
   ```
3. 或全量重建索引（先删后建）：
   ```bash
   curl -X DELETE 'http://localhost:9200/knowledge_base'
   docker exec rag-api python3 /app/scripts/index-docs.py /app/examples/aiops-docs/
   ```

## 七、常见问题

### Q: 服务名改了怎么办

1. 在 Neo4j 中更新 name 属性（`load-topology.py` 的 MERGE 语句自动处理）
2. 在文档 frontmatter 中更新 `service_name` 字段
3. 重新索引

服务 ID 不变，所有关联不中断。

### Q: 索引后搜索不到新文档

1. 检查 frontmatter 格式是否正确（YAML 语法错误会导致解析失败）
2. 确认 `doc_type` 是有效的（sop/tech/incident）
3. 检查 `service_id` 是否与 topology 中的一致
4. 查看 ES 索引总数：`curl 'http://localhost:9200/knowledge_base/_count'`

### Q: 文档太长会影响检索质量吗

文档入库时按 500 字分块（中文按句号/问号/感叹号/分号切割）。块越大越不精确但块数少，块越小越精确但块数多。当前 500 是经验值，可通过调 BM25 参数优化。

### Q: 如何验证拓扑关系正确

```bash
# 查单个服务
curl -s 'http://localhost:8001/api/v1/topology?service_id=svc_order' | python3 -m json.tool

# 查主机影响分析
curl -s 'http://localhost:8001/api/v1/impact?host_id=host_app_01' | python3 -m json.tool

# 查依赖链
curl -s 'http://localhost:8001/api/v1/graph/path?service_id=svc_nginx&depth=3' | python3 -m json.tool
```

## 九、ES ↔ Neo4j 同步维护

### 写入流程

文档索引时自动同步 Neo4j：
```
index-docs.py → ES 索引 chunk → upsert Neo4j Document 节点 → merge HAS_DOC 边
```

同步约束：
- `service_ids` 中的每个 Service 必须在 Neo4j 中存在，否则拒绝写入 + 报 warning
- 每个文档只同步一次（文档级，非 chunk 级）
- Neo4j 瞬时故障时 driver 内置 15s 指数退避自动恢复

### 同步健康检查

```bash
# 快速检查（/health 内含 sync 段）
curl -s http://localhost:8001/api/v1/health | python3 -m json.tool

# 详细一致性报告
curl -s http://localhost:8001/api/v1/health/sync | python3 -m json.tool
```

返回字段说明：
| 字段 | 含义 |
|------|------|
| `orphan_docs` | Neo4j 有 Document 节点但 ES 查不到 — 需清理 |
| `dangling_doc_refs` | ES 有文档但 Neo4j 无对应 Document 节点 — 需重新索引 |
| `missing_doc_edges` | HAS_DOC 边指向已删除的 Document 节点 |
| `cluster_sync_ok` | cluster_nodes 全部 host 引用有效 |

### 修复不一致

- **orphan_docs > 0**：Neo4j Document 节点残留。手动确认后删除
- **dangling_doc_refs > 0**：文档索引后 Neo4j 同步失败。重新跑 `index-docs.py`
- **missing_doc_edges > 0**：边未清理干净。重新索引即可

### 定时审计

`sync-health-check.sh` 每天 03:00 自动跑，输出报告到 `reports/` 目录：
- `reports/sync-health-YYYY-MM-DD.json` — 完整报告
- `reports/sync-health-summary.log` — 每日摘要
- `reports/sync-health-alerts.log` — 异常记录（orphan > 0 或 dangling > 0）

### Metrics 说明

新增 5 个监控指标：
| metric | 说明 |
|--------|------|
| `sync_errors_total` | Neo4j 文档同步失败次数 |
| `orphan_docs_total` | health/sync 最近一次扫描的孤儿文档数 |
| `degraded_queries_total` | Neo4j 不可用时降级查询次数 |
| `entity_extract_fallback_total` | LLM 实体提取失败时正则兜底次数 |
| `llm_tokens_total` | LLM API token 累计消耗 |
